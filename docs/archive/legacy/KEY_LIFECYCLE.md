# Key Lifecycle

Status: v5 design complete. Implemented baseline covers secret generation,
redaction, certificate pinning, credential rotation config validation, and a
redacted key inventory command. SDK profile metadata separates active and next
client credential secrets from serializable profile metadata through a
`ProfileSecretStore` contract, and `NativeProfileSecretStore` wires that
contract to the platform credential store for future macOS GUI use. Crypto
experiment keys are represented only as disabled registry entries in product
config; no HPKE, Noise, or ML-KEM product runtime keys are generated or
consumed. The feature-gated Noise core harness uses test static keys in local
tests only.

## Key and Secret Classes

- User credential secrets: generated with `maverick gen-user`, high entropy,
  redacted in Debug and Display, zeroized on drop.
- Server TLS private keys: supplied by operator, never generated by Maverick in
  normal deployment.
- Certificate pins: optional leaf certificate SHA-256 pins after normal TLS
  validation.
- Client next-credential secrets: optional local material for explicit
  `auth.rotation.auto_switch`; treated like active user credential secrets.
- Auth v2 epoch keys: derived transiently from user secrets when Auth v2 is
  explicitly enabled.
- Future experimental product keys: HPKE, Noise, or ML-KEM keys behind feature
  gates only.

## Lifecycle Rules

- Generate secrets with the CLI or an audited external secret manager.
- Store secrets only in local config or platform secure storage.
- GUI profile metadata must store only secret references; raw active and next
  credential secrets belong in a `ProfileSecretStore` implementation.
- Local tests must not write the user's real Keychain; use
  `InMemoryProfileSecretStore` for contract tests.
- Never print secrets, HMAC tags, shared secrets, or TLS key material.
- Rotate user credentials with bounded overlap windows.
- Remove expired previous credentials after migration.
- Revoke disabled users before considering credential rotation state.

## Operational Checklist

1. Generate a new high-entropy secret.
2. Add it as active or next credential according to the rotation plan.
3. Keep previous credentials bounded by `not_before` and `not_after`.
4. Validate config with `maverick check-config`.
5. Use `maverick migrate-config` dry-run before deployment.
6. Review configured credential material with
   `maverick key-inventory --kind client|server -c <config>`.
7. Run `maverick rotate-credential --server <config> --dry-run` to identify
   expired previous credentials and next credentials ready for promotion.
8. Remove expired credentials and verify logs are redacted.

## Redacted Inventory

`maverick key-inventory` prints only metadata about configured key and
credential material. It redacts active user secrets, client next-credential
secrets, previous credential secrets, TLS private-key contents, and full
credential ids. The command is intended for operator review and support
diagnostics, not for exporting usable credentials.

## Loss, Compromise, And Recovery

- Lost user credential: generate a replacement; never attempt to recover or log
  plaintext from redacted output.
- Compromised user credential: disable/revoke it, investigate privately, rotate,
  and verify the old credential fails.
- Lost TLS private key: issue a new key/certificate, update pins deliberately,
  and retire the old pin after the migration window.
- Lost release-signing private key: retain old public verification material,
  create a new signer, publish its fingerprint through a reviewed channel, and
  sign only future artifacts.
- Suspected release or APT key compromise: stop publication, identify the last
  trusted signed state, rotate the affected key, replace affected artifacts or
  metadata, and issue an advisory.
- Lost encrypted Linux service credential: import a new credential through the
  fixed root-only path. Do not weaken the credential-root mode or write plaintext
  to a config/environment file to restore service quickly.

Commit/tag, OpenSSH artifact, APT archive, TLS, and user credential keys are
separate trust classes and must not be reused. Full incident steps are in
`docs/INCIDENT_RESPONSE.md`.
