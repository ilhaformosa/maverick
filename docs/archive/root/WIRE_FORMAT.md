# Maverick Wire Format

Status: Auth v1/v2 frame fields are frozen for the narrow
`maverick-tls-h2-cli-v1` scope. Auth v2 and non-H2 carriers remain explicit or
experimental as documented. This is not a standardization claim.

Maverick frames are carried inside the encrypted tunnel request/response body.
The mandatory default carrier is HTTP/2; the v2 baseline can also carry the
same frames over feature-gated H3/QUIC. No Maverick magic value appears in
cleartext HTTP headers.

## Frame Header

Each frame is:

```text
frame_type: u8
flags:      u8
flow_id:    u64  network byte order
length:     u32  network byte order
payload:    length bytes
```

Header size is 14 bytes. `length` must be no larger than configured
`max_frame_size`, default 65536.

Malformed frames close the authenticated session. Parsers must reject malformed
input without panicking.

## Frame Types

```text
0x01 CLIENT_HELLO
0x02 SERVER_HELLO
0x03 OPEN_TCP
0x04 TCP_DATA
0x05 TCP_FIN
0x06 TCP_RESET
0x07 OPEN_UDP
0x08 UDP_PACKET
0x09 CLOSE_FLOW
0x0A PING
0x0B PONG
0x0C WINDOW_UPDATE
0x0D ERROR
0x0E DNS_QUERY
0x0F DNS_RESPONSE
0x10 PADDING
```

`ERROR` frames are only valid after authentication succeeds.

## CLIENT_HELLO Payload

```text
protocol_version: u16
client_nonce:     32 bytes
timestamp_unix:   i64
credential_len:   u16
credential_id:    credential_len bytes UTF-8
mode:             u8  0 auto, 1 stable, 2 private
feature_flags:    u64
auth_tag:         32 bytes HMAC-SHA256
```

## SERVER_HELLO Payload

```text
protocol_version_selected: u16
server_nonce:              32 bytes
session_id_len:            u8
session_id:                session_id_len bytes
max_frame_size:            u32
max_concurrent_flows:      u32
feature_flags_selected:    u64
server_auth_tag:           32 bytes HMAC-SHA256
```

## OPEN_TCP Payload

```text
address_type: u8  0x01 IPv4, 0x03 domain, 0x04 IPv6
address:      variable
port:         u16
initial_len:  u32
initial_data: initial_len bytes
```

Domain addresses are UTF-8 with a `u16` length prefix. IPv4 is 4 bytes and IPv6
is 16 bytes.

## TCP_DATA Payload

Raw TCP bytes.

## DNS_QUERY / DNS_RESPONSE Payload

Raw DNS packet bytes. The v1.1 implementation forwards one DNS packet per
authenticated HTTP/2 tunnel request.

## PADDING Payload

Opaque padding bytes generated by bounded shaping policy. Receivers ignore the
payload and do not forward it to application flows.

## OPEN_UDP Payload

```text
idle_timeout_ms: u64
```

`OPEN_UDP` starts an authenticated UDP relay flow. The server caps the requested
idle timeout to `server.advanced.udp_idle_timeout_ms` and acknowledges setup
with `WINDOW_UPDATE`. Either side may close the flow with `CLOSE_FLOW`.

## UDP_PACKET Payload

Uses the same target-address layout as `OPEN_TCP`, with the datagram bytes in
the trailing data field:

```text
address_type: u8  0x01 IPv4, 0x03 domain, 0x04 IPv6
address:      variable
port:         u16
data_len:     u32
data:         data_len bytes
```

The current client exposes this through SOCKS5 UDP ASSOCIATE. Fragmented SOCKS5
UDP packets are not supported.

## Reserved Fields

`flags` is reserved for future compatible extensions.

`feature_flags` is a `u64` extension bitmap. Bit `63`
(`FEATURE_TLS_CHANNEL_BINDING`) requests TLS exporter channel binding for the
auth HMAC. When selected, both ClientHello and ServerHello HMAC transcripts
include the 32-byte TLS exporter output labeled
`maverick tls channel binding v1`. The wire payload shape does not change.

Implementations must not select unsupported feature flags. Clients reject
ServerHello messages that select flags the client did not request.
